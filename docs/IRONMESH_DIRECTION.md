# IronMesh 发展方向

IronMesh 的长期方向不是做一套微服务组件拼装工具，而是成为一个 Rust 原生的分布式业务运行时。

它的核心气质应该是：集群控制面内生、数据面轻快、业务节点自治。业务服务不是挂在一堆外部中间件上，而是天然生长在一个有一致性、有拓扑、有控制面的系统里。

## 核心定位

| 方向 | 建议 |
|---|---|
| 系统定位 | IronMesh 是 Rust 原生分布式业务系统框架，不是组件胶水。 |
| 集群模型 | 以 Raft 控制面为核心，注册节点是强一致骨架，网关和业务节点是可扩展 learner。 |
| 通信模型 | TCP 长连接作为集群内部主干，HTTP 只做人工验证、管理入口或兼容出口。 |
| 状态模型 | 所有集群事实进入 `IronClusterData`，包括节点、服务、能力、负载、路由和版本。 |
| 读取模型 | 写入走 leader，读取尽量走本地快照，让网关和业务节点拥有本地集群视图。 |
| 扩容模型 | 节点启动后加入集群，服务能力写入状态机，网关自动感知并参与路由。 |
| 业务入口 | 优先支持游戏、实时业务、高并发网关和内部业务 RPC。 |

## 关键设计

| 设计 | 说明 |
|---|---|
| 集群事实层 | Raft 状态机只存集群事实，不做通用数据库。它记录节点是否存在、服务是否在线、服务能力、节点负载、路由版本和职责归属。 |
| 网关本地自治 | 网关不应该每次请求都查询注册中心。它应该从本地状态机快照读取服务拓扑，并基于本地视图完成路由选择。 |
| 业务节点集群化 | 业务服务不只是注册表中的一条记录，而是有身份、有连接、有状态、有拓扑位置的集群成员。 |
| 控制面强一致 | 节点加入、服务注册、能力变更、路由版本等少量关键写入进入 Raft，确保集群事实有统一顺序。 |
| 数据面高性能 | 实际业务流量不进入 Raft。数据面走轻量 TCP、业务协议和本地路由决策，保证吞吐和低延迟。 |
| 实验室验证 | `iron-zenith-lab` 应持续承载压测客户端、节点模拟器、故障注入、扩缩容验证和拓扑观察工具。 |
| 样板业务牵引 | 先用实时游戏大厅、房间服务、高并发网关或内部 RPC 网关验证框架价值。 |

## 演进重点

| 阶段 | 目标 |
|---|---|
| 控制面 API | 稳定节点状态、leader、membership、本地快照、服务注册和主动退出等 API。 |
| 业务数据模型 | 让 `IronClusterData` 从最小 KV 演进为节点、服务、能力、负载、路由和版本的明确模型。 |
| 服务发现 | 网关和业务节点通过本地状态机快照获得服务拓扑，减少集中查询和运行时依赖。 |
| 热扩容 | 新节点加入后写入服务实例，网关本地状态同步后自动参与路由。 |
| 负载路由 | 状态机记录负载摘要，网关结合本地连接状态和策略选择后端节点。 |
| 自动化验证 | 在 `iron-zenith-lab` 中覆盖节点加入、移除、leader 切换、写入转发、扩缩容和高并发场景。 |
| 生产边界 | 补齐认证鉴权、配置化、观测日志、协议版本和部署边界。 |

## 设计气质

| 原则 | 说明 |
|---|---|
| 少依赖外部中间件 | 优先让核心集群能力在 IronMesh 内部闭环。 |
| 少胶水，多内聚 | 核心能力围绕统一控制面演进，避免多个组件各自维护一套事实。 |
| 少黑盒，多可观察 | 节点状态、membership、写入路径、断线移除和路由决策都应该可日志化、可查询。 |
| 少配置魔法，多显式模型 | 重要行为由明确数据结构和 API 表达，不依赖隐式约定。 |
| 少大而全，多关键路径极强 | 先把集群控制面、TCP 通信、路由和扩容这几条关键路径做到扎实。 |
| Rust 原生 | 保持低资源占用、低延迟、强类型边界和可控运行时。 |

IronMesh 最好成为一个集群控制面内生、数据面轻快、业务节点自治的 Rust 分布式业务运行时。
