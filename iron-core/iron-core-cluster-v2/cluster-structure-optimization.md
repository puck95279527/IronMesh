# iron-core-cluster-v2 结构优化备忘

这份文档只记录一件事：后续优化 `iron-core-cluster-v2` 时，优先按照“阶段清楚、方法稳定、参数少、日志清楚”的方向收口，不要把启动逻辑继续切得过碎。

## 总原则

1. 入口少。
   - `run()` 只保留一条主线。
   - 主线只讲清楚“校验 -> 启动运行时 -> 发现/起盘 -> 加入节点 -> 保持运行”。

2. 阶段少。
   - 整个启动过程控制在 4 到 6 个稳定阶段。
   - 不要为了“看起来更细”而继续无限拆方法。

3. 方法内保持线性叙事。
   - 一个方法只做一类事。
   - 尽量遵循“先准备、再执行、最后确认”的顺序。

4. 参数少。
   - 能收成上下文结构体，就不要在方法之间反复传一长串参数。
   - 尽量减少 `node_id`、`node_name`、`node_addr`、`boot_nodes`、`raft`、`self_tag` 这类重复传递。

5. 日志做路标。
   - 每个阶段开头和结尾保留一条关键日志。
   - 让人能顺着日志直接读出集群启动链路。

## 推荐阶段

| 阶段 | 作用 |
|---|---|
| `validate_topology` | 校验当前节点和 `boot_nodes`。 |
| `build_raft_runtime` | 构造 Raft、TCP 服务和调试 HTTP 所需的运行时。 |
| `bootstrap_or_join_cluster` | 判断当前节点是加入已有集群，还是自己起盘。 |
| `join_remaining_nodes` | 在起盘成功后，把其他节点逐个加入。 |
| `spawn_debug_http` | 单独启动调试 HTTP 面。 |
| `serve_forever` | 保持进程运行，不让节点退出。 |

## 推荐文件职责

| 文件 | 职责 |
|---|---|
| `cluster_manager.rs` | 只负责流程编排。 |
| `cluster_runtime.rs` | 负责运行时搭建。 |
| `cluster_bootstrap.rs` | 负责起盘和自动加入。 |
| `cluster_join.rs` | 负责逐个加入其他节点。 |
| `cluster_http.rs` | 负责调试 HTTP。 |

## 需要记住的判断

- 不是方法越多越好。
- 不是方法越少越好。
- 最合适的是：方法数量适中、阶段明确、参数少、日志清楚、文件职责稳定。

## 以后优化时的提醒

- 如果某个方法里同时出现“判断、启动、回退、重试、日志”五类动作，优先考虑再收口。
- 如果一个文件里混进了太多阶段逻辑，优先考虑拆成更稳定的职责块。
- 如果你在调试时总是要在很多方法之间来回跳，说明这块还可以继续收成更清楚的阶段。
