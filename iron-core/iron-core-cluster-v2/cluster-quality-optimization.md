# iron-core-cluster-v2 质量与稳定性优化清单

本文档用于记录 `iron-core-cluster-v2` 在“不持久化、最小、简单、稳定”的目标下，后续需要逐步优化的点。

## 目标边界

| 项目 | 说明 |
|---|---|
| 不做持久化 | Raft log、vote、state machine、snapshot 继续使用内存实现。 |
| 不扩散功能 | 优先优化现有启动、TCP 通信、成员维护、日志和运行稳定性。 |
| 不自动移除 voter | `cluster-reg-*` 投票节点只探测和记录日志，不自动从 membership 移除。 |
| learner 自动移除 | 普通 learner 节点 TCP 不可达后，由 leader 确认并移出集群。 |
| `/raft/metrics` 保持真实 | 查询接口继续返回 OpenRaft 原始真实 metrics，不做展示层过滤。 |

## 优化总表

| 优先级 | 分类 | 优化项 | 当前状态 | 建议处理 | 原因 |
|---|---|---|---|---|---|
| P0 | 起盘安全 | 跨机器安全起盘机制 | 当前使用 `127.0.0.1:4999` 本地端口锁 | 需要优化 | 本地锁只能防同机多进程；跨机器多个 boot 节点同时启动时，仍可能各自起盘。 |
| P0 | TCP 安全边界 | TCP frame 最大 body 长度限制 | 当前读取长度后直接分配 `Vec` | 需要优化 | 异常长度可能导致大内存分配，影响节点稳定性。 |
| P0 | TCP 连接稳定性 | TCP read/write 超时 | 当前读写可能长期等待 | 需要优化 | 避免异常连接长期占用任务。 |
| P0 | TCP 并发控制 | TCP server 连接数限制 | 当前每个连接直接 `tokio::spawn` | 需要优化 | 防止连接过多导致任务无限增长。 |
| P0 | Raft 成员安全 | voter 不自动移除 | 当前已经做到 | 保持 | 避免自动移除 leader/follower 破坏多数派。 |
| P0 | Learner 清理 | learner 不可达后自动移除 | 当前已经做到 | 保持并观察 | 符合当前最小集群目标。 |
| P1 | 起盘流程 | 起盘前与抢锁后二次探测 | 当前已经做到 | 保持 | 能降低重复起盘风险。 |
| P1 | 起盘锁实现 | 起盘锁逻辑抽象化 | 当前写死本地 TCP 端口锁 | 需要优化 | 先让职责边界清楚，后续替换跨机器机制更容易。 |
| P1 | 后台任务 | `tokio::spawn loop` 生命周期管理 | 当前长期运行，无统一 shutdown | 可优化 | 当前不做优雅退出时可先保持；未来需要更明确的任务边界。 |
| P1 | 后台任务 | learner/voter 探测串行 | 当前串行探测 | 可优化 | 节点数量增加时，维护循环可能被拖慢。 |
| P1 | 日志质量 | voter 不可达日志节流 | 当前长期不可达时会重复记录 | 可优化 | follower 长期停止时可能刷 info 日志。 |
| P1 | 日志质量 | leader 切换日志 | 当前已经有日志 | 保持 | 停 leader 后，新 leader 会明确记录开始承担维护任务。 |
| P1 | HTTP 查询 | `/raft/metrics` 直接暴露 OpenRaft metrics | 当前直接返回 | 保持 | 便于查看真实集群关系。 |
| P2 | 网络协议 | JSON TCP RPC | 当前使用 JSON frame | 保持 | 简单直观，当前不是主要瓶颈。 |
| P2 | 网络协议 | 认证与鉴权 | 当前无认证 | 暂不做 | 当前目标是最小集群；后续跨机器生产化时再评估。 |
| P2 | 存储 | 内存 Raft log/state/snapshot | 当前使用内存存储 | 明确保持 | 不持久化是当前设计目标。 |
| P2 | 配置 | timeout 和 interval 常量化/配置化 | 当前部分常量化 | 可优化 | 先保持常量，后续需要多环境运行时再配置化。 |
| P2 | 测试 | 自动化集群场景测试 | 当前主要手动验证 | 可优化 | 稳定后可补集成测试，覆盖节点加入、移除、leader 切换。 |

## 优化主线

| 主线 | 包含优化项 | 建议推进方式 |
|---|---|---|
| 起盘安全线 | 跨机器安全起盘机制、起盘锁抽象化、二次探测保持 | 优先推进，这是当前分布式场景最大的真实风险。 |
| TCP 稳定线 | frame 长度限制、读写超时、连接数限制 | 第二优先级，先做边界保护，再考虑性能细节。 |
| 成员维护线 | learner 自动移除、voter 只日志、voter 日志节流 | 当前功能已成型，后续以日志节流和误判控制为主。 |
| 任务运行线 | spawn loop 生命周期、串行探测、错误边界 | 当前保持简单，等核心稳定后再收口。 |
| 观察诊断线 | metrics、日志文案、leader 切换日志 | 继续保持清楚、少而准，不引入复杂展示层。 |
| 明确不做线 | 持久化、复杂认证、复杂协议、自动移除 voter | 防止目标扩散，保持最小集群定位。 |

## 推荐推进顺序

| 顺序 | 优化项 | 预期结果 |
|---|---|---|
| 1 | 跨机器安全起盘机制 | 避免多机器 boot 节点同时自起盘。 |
| 2 | TCP frame 最大长度限制 | 防止异常帧导致内存风险。 |
| 3 | TCP read/write 超时 | 防止坏连接长期占用任务。 |
| 4 | TCP server 并发连接限制 | 控制连接和任务增长。 |
| 5 | voter 不可达日志节流 | 保持 leader 日志可读。 |
| 6 | 起盘锁职责抽象化 | 为后续替换锁实现做准备。 |
| 7 | learner/voter 探测并发优化 | 节点数量增加后提升维护效率。 |
| 8 | 自动化集群场景测试 | 固化当前行为，避免后续优化破坏主逻辑。 |
