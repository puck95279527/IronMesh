# iron-core-cluster-v2 质量与稳定性优化清单

本文档用于记录 `iron-core-cluster-v2` 在“不持久化、最小、简单、稳定”的目标下，已经完成、明确保持和后续需要逐步优化的点。

## 目标边界

| 项目 | 说明 |
|---|---|
| 不做持久化 | Raft log、vote、state machine、snapshot 继续使用内存实现。 |
| 不扩散功能 | 优先优化现有启动、TCP 通信、成员维护、日志和运行稳定性。 |
| 固定 Boot 起盘 | 配置中唯一 `is_boot_node = true` 的注册节点负责首次起盘。 |
| 不自动移除 voter | `cluster-reg-*` 投票节点只探测和记录日志，不自动从 membership 移除。 |
| learner 自动移除 | learner 节点 TCP 不可达后，由 leader 确认并移出集群。 |
| `/raft/metrics` 保持真实 | 查询接口继续返回 OpenRaft 原始真实 metrics，不做展示层过滤。 |

## 已完成与保持

| 分类 | 优化项 | 当前状态 | 说明 |
|---|---|---|---|
| 起盘安全 | 固定 Boot 节点起盘机制 | 已完成 | `is_boot_node = true` 的唯一节点负责首次 `initialize`，其它注册节点只等待加入。 |
| 起盘安全 | 移除 `4999` 本机起盘锁 | 已完成 | 启动安全不再依赖本地端口锁，避免跨机器语义混乱。 |
| 起盘流程 | 启动阶段整理 | 已完成 | `IronRaftClusterManagerFlow` 保持 6 个主阶段，注释按阶段排序。 |
| TCP 安全边界 | TCP frame 最大 body 长度限制 | 已完成 | 读取和写入 frame 前都会校验 `MAX_FRAME_BODY_LEN`。 |
| TCP 连接稳定性 | TCP frame read/write 超时 | 已完成 | frame 读、写、flush 都有统一超时。 |
| TCP 并发控制 | TCP server 连接数限制 | 已完成 | TCP 服务端通过 `Semaphore` 控制最大并发连接数。 |
| 配置维护 | 集中常量文件 | 已完成 | Raft、启动流程、TCP、join、维护任务常量集中在 `iron_raft_constants.rs`。 |
| 日志质量 | 启动日志文案 | 已完成 | 已去掉“争抢起盘锁”等旧语义，改为固定 Boot 起盘语义。 |
| Raft 成员安全 | voter 不自动移除 | 保持 | 避免自动移除 leader/follower 破坏多数派。 |
| Learner 清理 | learner 不可达后自动移除 | 已完成，继续观察 | 符合当前最小集群目标。 |
| 日志质量 | leader 切换日志 | 已完成，保持 | 新 leader 会记录开始承担集群维护任务。 |
| HTTP 查询 | `/raft/metrics` 直接暴露 OpenRaft metrics | 保持 | 便于查看真实集群关系。 |
| 网络协议 | JSON TCP RPC | 保持 | 简单直观，当前不是主要瓶颈。 |
| 存储 | 内存 Raft log/state/snapshot | 明确保留 | 不持久化是当前设计目标。 |

## 仍需优化

| 优先级 | 分类 | 优化项 | 当前状态 | 建议处理 | 原因 |
|---|---|---|---|---|---|
| P1 | 后台任务 | `tokio::spawn loop` 生命周期管理 | 长期运行，无统一 shutdown | 可优化 | 当前不做优雅退出时可先保持，后续可明确任务边界。 |
| P1 | 后台任务 | learner/voter 探测串行 | 当前串行探测 | 可优化 | 节点数量增加时，维护循环可能被拖慢。 |
| P1 | 日志质量 | voter 不可达日志节流 | 长期不可达时会重复记录 | 可优化 | follower 长期停止时可能刷 info 日志。 |
| P2 | 网络协议 | 认证与鉴权 | 当前无认证 | 暂不做 | 当前目标是最小集群；后续跨环境生产化时再评估。 |
| P2 | 配置 | timeout 和 interval 配置化 | 当前集中常量化 | 可优化 | 先集中常量，后续需要多环境运行时再配置化。 |
| P2 | 测试 | 自动化集群场景测试 | 当前主要手动验证 | 可优化 | 稳定后可补集成测试，覆盖节点加入、移除、leader 切换。 |

## 优化主线

| 主线 | 包含优化项 | 建议推进方式 |
|---|---|---|
| TCP 稳定线 | frame 长度限制、读写超时、连接数限制 | P0 已完成，后续只观察参数是否需要调整。 |
| 成员维护线 | learner 自动移除、voter 只日志、voter 日志节流 | 功能已成型，后续以日志节流和误判控制为主。 |
| 任务运行线 | spawn loop 生命周期、串行探测、错误边界 | 当前保持简单，等核心稳定后再收口。 |
| 观察诊断线 | metrics、日志文案、leader 切换日志 | 继续保持清晰、少而准，不引入复杂展示层。 |
| 明确不做线 | 持久化、复杂认证、复杂协议、自动移除 voter | 防止目标扩散，保持最小集群定位。 |

## 推荐推进顺序

| 顺序 | 优化项 | 预期结果 |
|---|---|---|
| 1 | voter 不可达日志节流 | 保持 leader 日志可读。 |
| 2 | learner/voter 探测并发优化 | 节点数量增加后提升维护效率。 |
| 3 | 自动化集群场景测试 | 固化当前行为，避免后续优化破坏主逻辑。 |
| 4 | timeout 和 interval 配置化 | 多环境运行时可以不改代码调参。 |
