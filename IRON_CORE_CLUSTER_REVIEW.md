# iron-core-cluster 评审报告

## 当前定位

`iron-core-cluster` 是 IronMesh 的集群控制面核心库，负责集群起盘、节点加入、Raft 选主、Raft TCP 通信、集群业务数据写入转发、本地状态读取和调试查询。

它服务于注册节点、网关节点和业务节点之间的集群协作，但不承载具体业务逻辑。

## 当前能力

| 类别 | 当前状态 | 评价 |
|---|---|---|
| 集群起盘 | 唯一 boot 节点初始化 Raft 集群 | 当前模型清晰，适合固定注册节点起盘 |
| 节点加入 | 注册节点加入为 voter，普通节点加入为 learner | 主流程已跑通 |
| Raft 通信 | Raft RPC 使用 TCP frame 传输 | 符合项目 TCP 优先边界 |
| 写入路径 | 非 leader 节点可转发写入到 leader | 基础写入链路可用 |
| TCP 连接 | 客户端复用 TCP 连接，服务端单连接可处理多请求 | 不是每次请求重新建连 |
| learner 移除 | learner TCP 复制连接断开后，由 leader 自动移除 | 已验证可用 |
| 调试查询 | HTTP 只暴露 `/health` 和 `/raft/metrics` | 适合人工验证，不作为业务 API |

## 已修复事项

| 类别 | 结论 |
|---|---|
| learner 自动移除 | 已恢复为基于 Raft TCP 复制连接失败事件触发，不做定时嗅探 |
| 断线事件投递 | 已改为等待投递，不再使用可能丢事件的 `try_send` |
| 移除失败处理 | `RemoveNodes` 失败后会短暂重试，最多 3 次 |
| voter 保护 | 只自动移除 learner，不自动移除 voter |
| 验证结果 | 本机启动 `reg1/reg2/reg3/gate5` 后，停止 `gate5`，leader 成功将节点 5 从 membership 移除 |
| 写入超时 | `client_write` 已使用独立业务写入超时，不再复用 join 超时 |
| TCP 空闲连接 | 等待下一帧头已增加空闲超时，避免空闲连接长期占用连接名额 |
| 启动加入流程 | boot 节点加入注册节点已从无限重试改为有限重试，失败后返回错误 |

## 仍需关注

| 编号 | 类别 | 问题 | 影响 | 评价 |
|---|---|---|---|---|
| 1 | 快照恢复 | 快照 membership 恢复时仍硬编码 `127.0.0.1:500{node_id}` | 跨机器或端口规则变化后，恢复出的节点地址可能错误 | 严重 |
| 2 | leader 切换写入 | 非 leader 写入失败后没有重新获取 leader 再重试 | leader 切换或 metrics 滞后时，业务写入可能失败 | 中高风险 |
| 3 | TCP 客户端并发 | 单目标连接由一个 mutex 串行读写 | 控制面可接受，高并发写入会排队 | 性能限制 |
| 4 | JSON 协议 | Raft RPC、快照、业务写入都走 JSON frame | 简单易调试，但不是高性能传输格式 | 当前可接受 |
| 5 | 本地状态读取 | `local_state_machine_data()` clone 整个状态机 | 数据量变大后会增加读路径开销 | 当前可接受 |
| 6 | 自动化验证 | `cargo test -p iron-core-cluster` 当前仍是 0 tests | 只能证明编译通过，不能证明集群行为稳定 | 需要补齐 |
| 7 | 对外错误表达 | 转发写入失败主要包装为 `std::io::Error` | 上层难以区分 leader 变化、超时、网络断开和远端拒绝 | 中风险 |

## learner 自动移除结论

当前 learner 自动移除已经按 TCP 事件驱动恢复。

leader 不做额外嗅探，也不按固定间隔扫描。它基于已有 Raft TCP 复制连接的失败事件判断 learner 断线，然后确认目标仍是 learner，再通过 Raft membership 变更移除该节点。

如果 learner 又重新启动，它应该重新走 join 流程加入集群。

## 生产级别结论

`iron-core-cluster` 当前已经能支撑本机和小规模 IronMesh 集群验证。集群起盘、节点加入、TCP 通信、写入转发和 learner 自动移除路径已经跑通。

它还没有完全达到生产级别。主要风险集中在快照地址恢复、leader 切换写入、自动化测试覆盖和部分接口表达。

下一步如果继续提高生产可靠性，优先处理“仍需关注”表中的严重和中高风险项。
